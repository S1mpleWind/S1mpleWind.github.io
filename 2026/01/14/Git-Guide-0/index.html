    Git 基础概念

    如果光看概念没有太明白也不用担心，跟着接下来的实践能更好地理解Git

    背景：集中式 vs 分布式

    • 集中式版本控制系统（CVCS）

      • 代表工具：SVN、CVS
      • 所有版本记录都保存在中央服务器上，开发者需要联网才能提交或更新代码。
      • 优点：管理集中，权限控制简单。
      • 缺点：依赖服务器，一旦服务器故障可能导致版本丢失；多人协作容易产生冲突。

    • 分布式版本控制系统（DVCS）

      • 代表工具：Git、Mercurial
      • 每个开发者的本地仓库都保存完整版本历史，操作大多在本地完成。
      • 优点：不依赖中央服务器、操作灵活、支持离线开发，容易进行分支和合并。
      • 缺点：初学者学习曲线稍陡，仓库可能体积大。

    Git 工作流程

    Git 的工作目录大致分为四层：

    1. 工作目录（Working Directory）

      • 就是你平常看到的项目文件夹
      • 可以修改、编辑、删除文件

    2. 暂存区（Staging Area / Index）

      • 临时存放即将提交的文件
      • 通过 git add 将文件从工作区放入暂存区

    3. 本地仓库（Local Repository）

      • 通过 git commit 将暂存区的内容保存到本地仓库
      • 每次提交都会生成一个唯一 ID

    4. 远程仓库（Remote Repository）

      • 位于远程服务器（如 GitHub、GitLab）
      • 通过 git push 上传本地提交，通过 git pull 获取远程更新

    本地的工作流程：主要在工作区、暂存区、本地仓库中进行
    最主要的内容为：

    • 将工作区的文件添加到暂存区；
    • 将暂存区的文件添加到仓库；
    • 从仓库中恢复不同版本的文件

    撤销更改&回滚历史记录

    git checkout、git reset、git revert

    在开发过程中，我们经常会遇到以下几类需求：

    1. 丢弃未保存的修改（代码写错了，不想要了）。
    2. 撤销已经 add 到暂存区的内容。
    3. 回到某个历史版本（彻底重置）。
    4. 撤销某次错误提交，但保留历史记录（更安全）。

    Git 提供了三种常用指令来处理：

    1. git checkout—— 丢弃工作区的修改

      1. 如果你改了某个文件，但还没执行 git add，
   可以用下面的命令恢复成最近一次提交的状态，
   这样文件会回到最后一次提交时的内容，未保存的修改会丢失。

        2.  而如果你已经执行了`git add `, 之后又修改了工作区的文件 ，
     工作区的文件会回到暂存区的状态

    也就是说checkout – file 的**本质是：用==暂存区覆盖工作区==**。

    如果暂存区里没有这个文件的改动，就用 HEAD（最近一次提交）的版本覆盖。

    1. git reset —— 回退到指定提交（可能丢历史） git reset 会改变当前分支的指向，让分支回到某个历史提交。
      它有三种常见模式：

    --soft

    仅回退分支指针到指定提交 , 保留暂存区和工作区的修改
    相当于 工作区、暂存区不变 ， 仓库删除指定提交后面的版本

    常用于：撤销提交，但想重新修改提交信息或合并到下一个提交

    --mixed（默认模式）

    回退分支指针，并清空暂存区

    工作区文件保留，但变为“未暂存状态”
    相当于 工作区不变，暂存区清空，仓库删除指定提交后面的版本

    常用于：取消 git add

    --hard

    回退分支指针，同时清空暂存区和工作区

    所有未保存的修改都会丢失

    常用于：彻底回退到某个版本

    1. git revert —— 撤销提交，但保留历史（推荐）

    git revert 会生成一个新的提交，用来“抵消”指定的提交。这样历史记录不会丢失。

    分支与合并

    在 Git 中，分支（branch） 是指向某个提交的可移动指针。
    它的作用是让我们在主线开发之外，开辟出一条独立的开发路线，用于开发新功能、修复 bug，而不会影响主分支的稳定性。

    1. 分支的概念及其重要性

    • 默认分支：新建仓库时会自动生成一个主分支（通常叫 main 或 master）。
    • 作用：通过分支，我们可以同时进行多条开发任务，比如一个分支写新功能，一个分支修 bug，互不干扰。

    创建PR(Pull Request)

    当你在GitHub上与团队协作开发项目时，Pull Request(PR) 是一个核心的协作机制。它不仅是代码变更的提交方式，更是团队代码评审和讨论的重要平台。

    PR的主要作用

    1. 代码合并请求：请求将你的分支（如feature/login）合并到目标分支（通常是main或master）
    2. 代码审查：团队成员可以查看你的变更，提出修改建议
    3. 持续集成：自动触发CI/CD流程，运行测试和构建
    4. 讨论平台：针对特定变更进行技术讨论

    创建PR的标准流程

    1. 准备工作：

      • 确保本地分支是最新的（git pull）
      • 解决所有代码冲突
      • 通过本地测试

    2. 推送分支：

    3. 创建PR：

      • 在GitHub仓库页面点击”New pull request”按钮
      • 选择正确的源分支(你的分支)和目标分支
      • 填写清晰的标题和描述（建议使用模板）

    4. 完善PR信息：

      • 关联相关issue（使用”Closes #123”语法）
      • 添加适当的标签（如bug, enhancement）
      • 指定审阅者(reviewers)

    5. 持续更新：

      • 根据review意见修改后，直接push到原分支
      • PR会自动更新，无需新建

    最佳实践

    • 保持PR规模适中（建议300行以内）
    • 一个PR只解决一个明确的问题
    • 提供清晰的修改背景和测试说明
    • 使用GitHub的预览功能检查变更

    Git 指令的基本范式

    Git 的命令和普通的bash命令一样，通常遵循以下结构：

    git <指令> [选项] [参数]

    • 指令（Command）：表示要执行的具体操作，例如 clone、commit、push。

    • 参数（Argument）：指令或选项所作用的对象，例如文件路径、分支名、提交 ID 等。

    常见问题与解决方案

    如何撤销错误的提交

    1. 撤销本地未推送的提交

      • 使用 git reset --soft HEAD~1 保留更改但撤销提交
      • 使用 git reset --hard HEAD~1 彻底丢弃更改
      • 示例：git reset --soft HEAD~1 && git commit -m "正确的提交信息"

    2. 撤销已推送的提交

      • 使用 git revert <commit-hash> 创建反向提交
      • 适用于共享仓库的撤销
      • 示例：git revert abc1234 然后推送反向提交

    3. 修改最近一次提交

      • git commit --amend 修改提交信息或内容
      • 适用于仅修改最后一次提交的情况

    如何恢复误删的分支

    1. 查找删除的分支记录

      • git reflog 查看所有操作历史
      • 查找删除前的commit记录

    2. 恢复分支方法

      • 通过commit hash恢复：git branch <branch-name> <commit-hash>
      • 示例：git branch feature/login abc1234

    3. 预防措施

      • 推送分支到远程：git push origin <branch-name>
      • 定期备份重要分支

    如何处理大文件或敏感信息

    1. 大文件处理方案

      • 使用Git LFS (Large File Storage)
      • 安装后配置：git lfs install
      • 跟踪大文件：git lfs track "*.psd"

    2. 移除已提交的敏感信息

      • 使用 git filter-branch 或 BFG Repo-Cleaner
      • 示例：bfg --replace-text passwords.txt my-repo.git
      • 完成后强制推送：git push --force

    3. 预防措施

      • 添加.gitignore文件排除特定文件
      • 使用环境变量存储敏感信息
      • 定期检查仓库中的大文件：git count-objects -vH# 常见问题与解决方案

    结语

    Git的核心价值总结

    Git作为分布式版本控制系统，其核心价值主要体现在以下几个方面：

    1. 版本控制能力：Git可以精确记录文件的历史变更，包括谁在什么时候修改了什么内容，形成了完整的开发历史记录。

    2. 分布式架构：每个开发者都拥有完整的仓库副本，可以在离线状态下工作，大大提高了开发灵活性和可靠性。

    3. 分支管理优势：Git的轻量级分支模型使得创建、合并和管理分支变得极其简单，支持高效的并行开发流程。

    4. 数据完整性保障：采用SHA-1哈希算法确保所有提交和文件内容都是不可篡改的，为代码提供了可靠的安全保障。

    5. 高效性能：大多数操作都在本地完成，响应速度快，能够轻松处理大型项目和复杂的历史记录。
